# Codex 编排工作流(通用规则)

把 codex 当作"实现者"、编排助手(如 Claude)只做分析与方向控制的一套通用工作规则。codex 通过本仓库的 MCP server `@yuemingruoan/codex-mcp-server` 调用。把本文件复制到你的全局或项目 `CLAUDE.md` 即可套用。

## 接入

```bash
claude mcp add codex -- npx -y @yuemingruoan/codex-mcp-server
```

前提:本机已安装并登录 Codex CLI(命令行 `codex` 或 Codex.app 桌面版均可,server 会自动探测;也可用环境变量 `CODEX_EXECUTABLE` / `CODEX_BIN` 显式指定二进制路径)。

## 职责边界:codex 做实现,编排助手只做分析 / 汇总 / 方向
- **编排助手只负责**:汇总与分析、根因判断、路线与方案设计、控制方向、审核 codex 产出、与用户沟通。不亲自写实现代码、不亲自跑命令、不亲自啃日志。
- **实现类工作一律交给 codex**:具体编码、改文件 / 机械重构、执行命令(构建 / 测试 / git / 安装 / 批量重跑等)、日志解析、grep / dump、复现实验。
- **可自己做的少数例外**:与用户沟通、方案 / 路线决策、审核 diff、改 `CLAUDE.md` / MCP 配置这类元 / 方向性小改、codex 不可用时的应急。

## 派发纪律(约束从严)
1. **先确认 MCP 正常**:`ping` + `version`,再跑一次有界执行确认能跑通,然后才派正式任务。
2. **任务小而明确,"三件套"齐全**:① 目标(要什么结果)② 执行方式 / 步骤(改哪些文件、跑哪些命令)③ 汇报形式(回传 diff / 关键结论 / 指定格式)。不给宽泛 / 开放式大目标——一轮塞太多会等很久还没效果、甚至返回空;宁可小步快跑、有界可验收。
3. **大任务必须拆分**:codex 上下文窗口较小,吃不下大而全的任务。拆成多个有界子任务,用 `batch-codex` 或多次并行 / 串行调用,每个子任务都要有独立、可验收的产出。
4. **codex 不可用时**:直接中止并向用户报告,不要私自改用其它手段硬干替代。

## 上下文隔离(保护编排能力)
- 派 codex 干活的目的之一,就是把实现细节挡在编排助手的上下文之外。要求 codex 只回传**精炼的结构化结论**(结论 / diff 摘要 / 关键数字 / 指定字段),不要原始大日志、整文件 dump、海量输出。
- 编排助手不亲自把大日志 / 大文件吞进上下文——让 codex 先消化,只取需要的那部分,避免细节冲淡上下文、削弱统筹全局的能力。

## 调用约定
- **full-access 发**:`sandboxMode: danger-full-access`(或 `yolo: true`)。
- 纯调查类任务:用 full-access 沙箱 + 在 prompt 里明确要求 codex 只读分析、不要修改文件。
